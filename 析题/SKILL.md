# Skill: 析题

## Keywords

题解、解题报告、启发式题解、题目分析、竞赛题解、比赛题解、AC题解、AtCoder题解、CF题解、代码精讲、详细题解

## Summary

为竞赛题目生成启发式题解。不拘模板，因题制宜——简单者一笔带过，复杂者层层剥茧。核心是「教人思考」，而非「给答案」。

## 触发条件

- 用户明确要求写「题解」「解题报告」「题目分析」
- 用户给了题面（或URL/文件），并期望得到思路解读
- 用户给了标程文件，希望逐行讲解

## 不触发的场景（边界）

- 用户只要求「搬运题目」「翻译题面」「打包题目」→ 这是搬题姬的活
- 用户只要求「生成测试数据」→ 这是搬题姬的活
- 用户说「这题怎么做」但没有题面也没有上下文 → 先确认题面再动手

## Strategy

### 第一步：获取题面

- 有 URL → urlgo 打开 snapshoot 获取（urlgo 不可用时用 WebFetch）
- 有文件路径 → Read 读取
- 用户直接在对话中给了题面 → 直接用
- 用户给了标程文件路径 → Read 读取，融入精讲

### 第二步：判断题型，决定深度

```
简单题（ABC的A/B，分值 ≤ 200）：
  → 简明扼要，一两段说清思路即可
  → 不需要「逐步推演」，不需要常见错误表
  → 代码精讲可精简到只标注关键行

中等题（ABC的C/D，分值 300~400）：
  → 展示核心思维过程
  → 有公式的地方用 LaTeX
  → 附常见错误表

难题（ABC的E/F/G，分值 ≥ 475）：
  → 充分展开：暴力 → 撞墙 → 优化 → 数据结构选型
  → 充要条件论证
  → 代码逐行精讲
  → 常见错误表 + 复杂度拆解
```

### 第三步：撰写

不给模板。以下只是方向性指引，非必须遵循：

- **从直觉出发**：读完题的第一反应是什么？哪怕是最笨的暴力。
- **展示歧路**：可能会想到的错误解法，以及为什么错。
- **关键转折**：什么观察让你找到了正确的路？
- **标程精讲**：若用户提供了标程文件，读取并逐关键行解释。
- **常见陷阱**：若有容易踩的坑，列表提醒。
- **多题串讲**：若是比赛题解，末尾总结各题之间的思维递进关系。

## 写作原则

```
┌─────────────────────────────────────────────┐
│  析题写作原则                                │
├─────────────────────────────────────────────┤
│  1. 启发优先：先给读者想的机会，再揭晓答案    │
│  2. 因题制宜：简单题不啰嗦，难题不敷衍        │
│  3. 展示过程：不只说"怎么做"，要说"怎么想"    │
│  4. 代码有魂：不贴裸代码，关键行必有注        │
│  5. 样例不篡改：题面样例原样保留              │
│  6. 公式用 LaTeX                             │
└─────────────────────────────────────────────┘
```

## AVOID

- ❌ AVOID 先给最优解再解释——读者不知道你为什么想到它
- ❌ AVOID 用「显然」「易知」「不难发现」跳过关键推理
- ❌ AVOID 对所有题用同样的深度——100分的题不值得写500字
- ❌ AVOID 篡改样例（特别是 AtCoder 的比赛禁令期间）
- ❌ AVOID 代码不加注释就贴上来
- ❌ AVOID 忘记给出复杂度分析
