## 📚 核心理念（来自 agent-skills）

### 1. 技能即上下文

```
┌─────────────────────────────────────────────────────────────┐
│  技能 = 预定义上下文 + 执行规则 + 验证标准                    │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  传统方式：用户描述 → AI 理解 → AI 执行（理解损耗大）        │
│  技能方式：用户触发 → 加载上下文 → 按规则执行（零损耗）      │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 2. 技能三要素

| 要素 | 说明 | 示例 |
|------|------|------|
| **触发条件** | 什么时候激活技能 | 关键字、场景、文件类型 |
| **执行规则** | 具体怎么做 | 步骤、模板、检查清单 |
| **验证标准** | 怎么判断做对了 | 输出格式、质量检查 |

### 3. 技能分层架构

```
skills/
├── {技能名}/
│   ├── SKILL.md              # 技能定义（必须）
│   ├── modules/              # 子模块（可选）
│   │   ├── 01-xxx.md         # 按步骤拆分
│   │   └── 02-xxx.md
│   ├── templates/            # 模板文件（可选）
│   │   └── example.md
│   └── examples/             # 示例文件（可选）
│       └── demo.md
```

## 🔧 技能设计流程

### 第一步：识别可复用模式

**触发信号**：
- 用户首次要求某类任务
- 判断将来可能重复
- 任务有明确的输入/输出/流程

**判断标准**：
```
✅ 值得沉淀成技能：
  - 用户工作场景中会经常遇到
  - 任务有明确的输入/输出/流程
  - 执行规则可以标准化
  - 验证标准可以量化

❌ 不值得沉淀：
  - 一次性任务
  - 流程不明确
  - 无法标准化
```

### 第二步：设计技能结构

**SKILL.md 必须包含**：

```yaml
---
priority: 500-1000           # 优先级（越高越先匹配）
name: SkillDesign
description: "一句话描述"     # 用于触发匹配
metadata:
  version: "1.0.0"
  trigger: "触发场景描述"
  copaw:
    emoji: "🎯"
    requires: {}             # 依赖的其他技能
    auto_load: true
---

# 技能名 - 功能描述

## 触发条件
- 关键字：xxx、yyy
- 场景：xxx

## 执行流程
1. 步骤一
2. 步骤二
3. 步骤三

## 输出格式
[模板或示例]

## 验证标准
- [ ] 标准 1
- [ ] 标准 2

## 示例
[完整示例]
```

### 第三步：命名原则

| 类型 | 示例 | 说明 |
|------|------|------|
| ✅ 好 | 搬题姬、墨池、雀影 | 意象化、拟人化、2-4字 |
| ✅ 好 | 破晓、格知、见自己 | 有意境、朗朗上口 |
| ❌ 差 | XXX工具、XXX助手、XXX生成器 | 功能性、无辨识度 |
| ❌ 差 | 超长名字、英文混杂 | 难记忆、难传播 |

### 第四步：验证技能质量

**检查清单**：
```
┌──────────────────────────────────────────────────────────────┐
│  ✅ 技能质量检查                                              │
├──────────────────────────────────────────────────────────────┤
│  □ 触发条件是否明确？                                         │
│  □ 执行流程是否可操作？                                       │
│  □ 输出格式是否有模板？                                       │
│  □ 验证标准是否可量化？                                       │
│  □ 是否有完整示例？                                           │
│  □ 名字是否简洁优雅？                                         │
│  □ 描述是否准确？                                             │
└──────────────────────────────────────────────────────────────┘
```

## 📖 技能设计最佳实践（来自 agent-skills）

### 1. Spec-Driven Development（规格驱动开发）

**核心理念**：先定义规格，再执行实现

```
用户需求 → 定义规格（Spec） → 验证规格 → 实现规格 → 验证实现
```

**Spec 文件结构**：
```markdown
# Spec: {功能名}

## 目标
[一句话描述要实现什么]

## 输入
- 参数 1: 类型、约束
- 参数 2: 类型、约束

## 输出
- 返回值: 类型、格式

## 约束
- 约束 1
- 约束 2

## 验收标准
- [ ] 标准 1
- [ ] 标准 2
```

### 2. Context Compression（上下文压缩）

**问题**：技能文件太大会占用上下文

**解决方案**：
```
SKILL.md（入口）→ 按需加载 modules/ → 用完释放
```

**原则**：
- SKILL.md 保持简洁（< 200 行）
- 详细步骤放 modules/
- 按步骤加载，不一次性全读

### 3. Tool Selection（工具选择）

**核心理念**：明确指定用什么工具

```yaml
allowed-tools:
  - Read        # 读文件
  - Write       # 写文件
  - Edit        # 编辑文件
  - Bash        # 执行命令
  - Browser     # 浏览器操作
```

**好处**：
- 减少工具选择错误
- 提高执行效率
- 降低出错概率

### 4. Verification-First（验证优先）

**核心理念**：先定义怎么验证，再开始实现

```
定义验收标准 → 实现功能 → 验证 → 通过/打回
```

**验收标准格式**：
```markdown
## 验收标准

### 功能验收
- [ ] 核心功能正常
- [ ] 边界情况处理

### 质量验收
- [ ] 代码格式规范
- [ ] 注释完整

### 性能验收
- [ ] 响应时间 < X秒
- [ ] 内存占用 < XMB
```
