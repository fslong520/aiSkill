## 核心理念：好的讲义长什么样？

开发阶段不是"照着模板填内容"。AI 的创造力应该被用来**根据课程内容本身选择最合适的表达形式**。下面是一些衡量"好的教学材料"的质量维度，而非必须遵守的步骤。

### 一、好的讲义有灵魂——不是知识点堆砌

| 坏的样子 | 好的样子 |
|---------|---------|
| 知识点罗列："整型有int、short、long long……" | 有叙事线索："为什么计算机里的整数有范围？" |
| 套话开场："今天我们来学习××" | 有钩子："猜猜下面这个程序结果是什么？" |
| 每课结构一模一样 | 根据内容选择最合适的结构 |

**怎么做由你决定。** 举几个例子：
- 语法课可以用"问题→错误→修复"的叙事
- 算法课可以用"暴力→优化→进阶"的递进
- 思维课可以用"谜题→工具→应用"的框架

### 二、好的课堂学生有参与——不是老师从头讲到尾

讲了什么不重要，学生脑子里留下了什么才重要。

检验方式：**每讲完一个知识点，停下来让学生做点什么。** 形式随意：
- 手动模拟、填空推导、口头问答、选择题、代码填空、猜输出……
- 抽象概念重引导（手动模拟），直观概念重检验（选择题/问答）
- **原则：每个知识点后必有一个练习，但形式不拘**

### 三、好的例题学生能自己读——不需要老师逐字翻译

例题是给学生看的，不是给老师看的。好的例题：
- 题目描述清晰，格式整齐
- 输入输出样例对照
- 有 OJ 链接，方便学生提交

常见做法是两列布局（左题面右样例），参见 [references/例题模板规则.md](../references/例题模板规则.md)。这不是硬性要求——你可以根据内容选择更适合的呈现方式。

### 四、好的代码学生能看懂——但不是全抄

代码示例的核心矛盾：**给完整代码学生会抄，不给又不会写。**

解法：只给关键骨架，核心逻辑留空。注释写"为什么"而不是"是什么"。

```cpp
// ❌ 全给
int sum = 0;
for (int i = 1; i <= n; i++) sum += i;

// ✅ 留空
int sum = 0;
// 从 1 加到 n，怎么写？
for (______; ______; ______) {
    ______;
}
```

### 五、好的作业有梯度——不是所有人做同一道题

学生水平参差，一刀切的作业必然有人吃不饱、有人跟不上。

建议分层：
- **基础**：课上讲过原题或简单变形（保底）
- **提高**：需要综合运用（主力）
- **挑战**：超出课堂范围，供学有余力者（拔高）

具体题量和分层方式，看课程需要。不存在"必须基础3提高3挑战2"的铁律。

**MPE 兼容**：生成的 Marp 讲义需适配 Markdown Preview Enhanced（MPE）渲染。注意：
- 使用标准 Marp 前置元数据（`marp: true`, `theme: zg_green` 等）
- 避免 MPE 不支持的扩展语法
- class 和 div 自定义样式需在 MPE 中测试通过
- **超链接用 HTML 语法**：`<a href="url" target="_blank">文字</a>`，确保在新标签页打开，勿用 Markdown 的 `[文字](url)` 语法

## 开发阶段质量自检

不是"逐项打勾"，而是**凭直觉回答这三个问题**：

1. **学生上完这节课能做什么？** 说不清就说明目标不明确。
2. **这节课有没有一个让人记住的点？** 如果没有，那就只是信息流过了。
3. **最弱的学生能跟到哪一步？最强的学生能走多远？** 分层不是形式，是真照顾。

## 讲义文件处理规则（技术细节）

更新数据库时（`update_item.py`）：
1. **HTML 优先**：同时有 `.html` 和 `.md` 只添加 HTML
2. **MD 过滤**：`.md` 文件必须以 `_ppt` 结尾才会被添加
3. **支持类型**：`.pdf`、`.html`、`.md`
4. **Marp 条件**：仅当用户要求 PPT 格式时才用 Marp + `zg_green` 主题；默认输出普通 Markdown

## 操作优化要点

基于实际备课经验，总结操作要点，避免常见错误。

### 📋 任务开始前确认清单

1. **明确具体操作**
   - **错误案例**：用户说"把七和八融合"，理解成"三和七"、"七和八"反复出错
   - **正确做法**：
     - 用户说"七和八" → 立即确认："您指的是章节七（数组比较）和章节八（直方图）吗？"
     - 用户说"合并a到b" → 立即确认："是将a中的XX章节合并到b中的XX位置吗？"
     - **不要猜测，不要自信，先确认再动手！**

2. **检查内容重复性**
   - **错误案例**：a中的"直方图"和b中的"趣味统计"都是桶计数法，内容重复
   - **正确做法**：
     - 合并前，先对比两个章节的知识点
     - 如果发现重复，立即询问用户："这两个章节内容相似，是删除还是合并？"
     - 避免重复内容出现在最终讲义中

3. **确认文件命名风格**
   - **错误案例**：自作主张重命名文件，但命名风格可能不符合用户期望
   - **正确做法**：
     - 询问用户："这个新讲义叫什么名字？要俏皮一点（如'巧用数组当桶使'）还是严肃一点（如'一维数组进阶应用'）？"
     - 查看现有文件的命名风格，保持一致性

### 🔧 操作执行阶段

4. **小步修改，实时验证**
   - **错误案例**：一次性大范围替换，结果替换错了内容
   - **正确做法**：
     - 每次只改一个章节（如只改章节编号）
     - 修改后，立即读取文件确认是否正确
     - 使用 TaskCreate 记录每一步，完成一项标记一项

5. **保留备份，防止误操作**
   - **正确做法**：
     - 重要修改前，先备份文件：`cp file.md file.md.bak`
     - 如果出错，可以立即恢复
     - 批量操作前，先在小范围测试

6. **实时与用户确认**
   - **正确做法**：
     - 每完成一个大步骤（如删除章节、重命名文件），询问用户："这样对吗？"
     - 不要等所有操作完成后才让用户检查

### ✅ 完成后检查阶段

7. **章节编号连续性检查**
   - **错误案例**：章节编号变成"一、三、四、五..."（跳过了二）
   - **正确做法**：
     - 修改章节后，立即检查编号是否连续
     - 子章节编号也要检查（1.1, 1.2, 1.3...）
     - 使用 Grep 工具查找所有章节标题，快速检查

8. **目录和链接同步更新**
   - **错误案例**：内容改了，但目录和 footer 中的链接还是旧的
   - **正确做法**：
     - 修改内容后，立即更新目录页（`CONTENTS`）
     - 更新 footer 中的页码链接（如 `[删除](#4)`）
     - 生成 PPT 后，验证所有链接是否正确跳转

   **⚠️ Marp 锚点计算规则（重要）：**
   Marp 中锚点 `#n` 指向第 n 张幻灯片，**必须通过 `---` 分隔符计数**，而非猜测编号：
   - 第一个 `---` 到第二个 `---` 之间是 YAML front matter（不计入幻灯片）
   - 第二个 `---` 之后开始算幻灯片 1
   - 每出现一个 `---` 即开始一张新幻灯片
   - 即：第 n 个用于分隔的 `---` 之后是第 (n-1) 张幻灯片
   - **目录中的锚点编号需重新核算**，不可直接沿用之前的编号（修改后幻灯片数量变化，锚点全变）
   - footer 中的导航锚点应与目录保持一致，数量不宜过多（5~7 个为宜）

9. **内容去重和完整性检查**
   - **正确做法**：
     - 检查是否有重复的知识点
     - 检查是否有冗余的示例
     - 确保每节课内容量适中（不要太少或太多）

### 🚨 备课常见错误与避免方法

| 错误类型           | 表现                            | 避免方法                                   |
| ------------------ | ------------------------------- | ------------------------------------------ |
| **理解错误**       | 把"七和八"理解成"三和七"        | 重复确认章节编号，不要自信猜测             |
| **错误添加内容**   | 从源文件复制了不该复制的内容    | 先明确"要什么"，而不是"不要什么"           |
| **编号混乱**       | 章节编号不连续（一、三、四...） | 每次添加/删除章节后，立即检查并修复编号    |
| **链接未同步**     | 目录和 footer 中的链接还是旧的  | 修改内容后，立即更新目录和 footer          |
| **内容重复**       | 相似知识点出现在多个章节        | 操作前先对比内容，合并重复部分             |
| **命名不符合期望** | 自作主张重命名，但风格不对      | 先询问用户命名风格，查看现有文件保持一致性 |
